# Security Design / 技术防护说明

> **简体中文版本说明**  
> 本文档使用较中性的简体中文（zh-CN）术语整理。不同社群、地区与服务器可能有不同规则和语境；实际部署时，请根据自己的服务器规则、Discord 平台规范与适用法律进行调整。

本文档说明 Discord OLLAMA BOT 目前采用的安全设计、风险降低措施与限制。

本项目包含 Discord 管理辅助、违规检测、案件报告、本地 AI、音乐播放、PC 状态报告等功能。由于涉及服务器管理与用户数据，用户在部署前应了解本项目的数据保存方式与安全限制。

---

## 1. 敏感数据不写死在代码中

本项目透过 `.env` 管理敏感设置，例如：

* Discord Bot Token
* Webhook URL
* 频道 ID
* 身份组 ID
* Ollama API 地址
* 音乐与 PC 状态功能设置

正式使用时请创建自己的 `.env`，并且不要将 `.env` 上传到 GitHub 或任何公开位置。

公开 repository 中只应提供：

```text
.env.example
```

用于示范设置格式，而不是保存真实 token 或私密数据。

---

## 2. Runtime Data 与源代码分离

本项目运行时会产生：

```text
data/
logs/
```

这些文件夹可能包含：

* 用户 ID
* 服务器 ID
* 频道 ID
* 警告记录
* 案件记录
* AI 互动记录
* 音乐库数据
* 错误记录

这些数据应保留在本机或私有環境中，部署者可以依需要检查、备份或清理。

---

## 3. 违规检测采用规则分类

违规检测规则拆分为多个分类，例如：

* `blocked_terms`
* `english_abuse_terms`
* `sexual_terms`
* `spam_terms`
* `target_patterns`
* `safe_phrases`

这样可以降低单一黑名单过于宽泛造成的误判。

其中：

```text
target_patterns
```

用来辅助判断消息是否可能针对某人。

```text
safe_phrases
```

用来排除教学、引用、代码示例、功能测试等场景，降低误判机率。

---

## 4. 误判排除机制

Bot 支援 safe phrase 机制。

当管理员确认某些内容是误判时，可以加入例外词，让后續相同或相近场景降低严重度。

适合加入 safe phrase 的场景：

* 代码示例
* 教学内容
* 规则引用
* 测试消息
* 文件名或指令参数
* 非针对他人的讨论

---

## 5. 案件系统与人工确认

当 Bot 检测到疑似违规内容时，会创建案件并送到管理报告频道。

若启用：

```env
CREATE_CASE_THREADS=true
```

Bot 会为案件创建 Discord 讨论串，方便管理员集中讨论同一个案件。

建议流程：

```text
检测到疑似违规
↓
创建案件
↓
发送到管理频道
↓
创建讨论串
↓
管理员确认场景
↓
使用 /case_ignore、/case_resolve、/punish 等指令处理
```

这样可以避免 Bot 直接按照单次文字判断做出过度处罚。

---

## 6. 高风险操作需要管理员确认

涉及高风险操作时，例如：

* 禁言
* 踢出
* 封禁
* 删除消息
* 遠端电腦动作
* Bot 关闭

应由管理员确认后运行。

部分指令使用 `confirm:true` 才会真的运行。
如果 `confirm:false`，通常只会 dry-run 或显示建议。

---

## 7. 管理员与普通成员权限分离

本项目将普通成员指令与管理员指令分开。

普通成员可使用：

* `/status`
* `/ai`
* `/my_warnings`
* `/my_profile`
* 音乐相关指令

管理员可使用：

* 警告与处罚指令
* 案件处理指令
* 规则管理指令
* PC 状态与遠端动作指令
* Bot 模式与高温保护设置

管理员判断可按照 Discord 权限或 `.env` 中设置的 `ADMIN_ROLE_ID`。

---

## 8. public / private 回覆控制

部分 Slash Command 支援 `public` 参数。

```text
public:false
```

代表只让用户自己看到。

```text
public:true
```

代表公开显示在目前频道。

这可以避免警告记录、AI 摘要、用户状态等资讯被不必要地公开。

---

## 9. Log 截断与错误保护

Bot 会记录指令、错误与系统事件，但 log 写入时会限制单筆数据长度，避免过长内容造成 log 爆量。

Logger 本身也採取 best-effort 设计。
如果 log 写入失败，不应让整个 Bot 中断。

---

## 10. AI 功能的限制与隔离

AI 功能使用本地 Ollama。

需要注意：

* AI 回覆仅作为辅助建议
* 不应把 AI 判断当作唯一管理依据
* 高风险操作仍需管理员确认
* 用户互动摘要可能包含个人数据，应避免公开

Bot 也提供不同模式：

```text
normal
eco
emergency
```

在 `eco` 或 `emergency` 模式下，可以降低或停用 AI 与重负载任务，保留基本管理与报告功能。

---

## 11. 音乐文件限制

音乐功能会限制可接受的音频文件扩展名，例如：

```text
.mp3
.wav
.flac
.ogg
.m4a
```

并可透过 `.env` 设置：

```env
MUSIC_MAX_TRACKS_PER_USER=20
MUSIC_MAX_FILE_MB=50
```

用来限制每位用户的歌曲数量与单一文件大小。

所有上传音频文件的来源、授权与使用权由上传者自行确认并负责。

Bot 仅提供音频文件队列与播放功能，不代表服务器、管理员或 Bot 拥有、审核或授权该音频文件内容。

---

## 12. PC 状态与高温保护

PC 状态功能可报告：

* CPU 使用率
* RAM 使用量
* 磁碟使用量
* 网路流量
* GPU 状态
* CPU 温度

CPU 温度读取可搭配 LibreHardwareMonitor。

Bot 可设置高温保护阈值，例如：

```text
eco_temp
emergency_temp
shutdown_temp
```

当温度过高时，Bot 可以自动切换低功耗模式或紧急模式，必要时关闭 Bot 自己。

---

## 13. 已知限制

本项目仍有以下限制：

* 规则式违规检测可能误判
* AI 摘要可能产生不完整或不准确的判断
* Discord 权限设置错误可能導致指令無法正常运行
* Webhook URL、Bot Token 一旦外洩，需要立即更换
* 实际安全性仍取决于部署者的 `.env`、Discord 权限、服务器设置与数据管理方式

---

## 14. 建议部署方式

正式部署前建议：

1. 先使用 私有 Repository 测试
2. 确认 `.env` 沒有被 commit
3. 确认 `data/` 与 `logs/` 不会上传
4. 使用测试 Discord 服务器驗证功能
5. 开启必要权限，不要给 Bot 过高权限
6. 管理指令只开放给可信任角色
7. 定期检查 log 与案件记录
8. 若公开项目，请先清除所有私密设置与本机路径

---

## 15. 总结

本项目的安全设计重点不是完全自动化处罚，而是：

```text
检测 → 报告 → 讨论 → 人工确认 → 运行
```

透过案件系统、讨论串、safe phrase、权限分离、private 回覆、log 截断与 `.env` 分离，降低误判、数据外洩与过度处罚的风险。
