# Security Design / 技術防護說明

> **繁體中文版本說明**  
> 本文件使用較中性的繁體中文（zh-TW）用語整理。不同社群、地區與伺服器可能有不同規則與語境；實際部署時，請依自己的伺服器規則、Discord 平台規範與適用法律調整。

本文件說明 Discord OLLAMA BOT 目前採用的安全設計、風險降低措施與限制。

本專案包含 Discord 管理輔助、違規偵測、案件回報、本地 AI、音樂播放、PC 狀態回報等功能。由於涉及伺服器管理與使用者資料，使用者在部署前應了解本專案的資料保存方式與安全限制。

---

## 1. 敏感資料不寫死在程式碼中

本專案透過 `.env` 管理敏感設定，例如：

* Discord Bot Token
* Webhook URL
* 頻道 ID
* 身分組 ID
* Ollama API 位址
* 音樂與 PC 狀態功能設定

正式使用時請建立自己的 `.env`，並且不要將 `.env` 上傳到 GitHub 或任何公開位置。

公開 repository 中只應提供：

```text
.env.example
```

用來示範設定格式，而不是保存真實 token 或私密資料。

---

## 2. Runtime Data 與原始碼分離

本專案執行時會產生：

```text
data/
logs/
```

這些資料夾可能包含：

* 使用者 ID
* 伺服器 ID
* 頻道 ID
* 警告紀錄
* 案件紀錄
* AI 互動紀錄
* 音樂庫資料
* 錯誤紀錄

這些資料應保留在本機或私有環境中，部署者可以依需要檢查、備份或清理。

---

## 3. 違規偵測採用規則分類

違規偵測規則拆分為多個分類，例如：

* `blocked_terms`
* `english_abuse_terms`
* `sexual_terms`
* `spam_terms`
* `target_patterns`
* `safe_phrases`

這樣可以降低單一黑名單過度寬泛造成的誤判。

其中：

```text
target_patterns
```

用來輔助判斷訊息是否可能針對某人。

```text
safe_phrases
```

用來排除教學、引用、程式碼範例、功能測試等情境，降低誤判機率。

---

## 4. 誤判排除機制

Bot 支援 safe phrase 機制。

當管理員確認某些內容是誤判時，可以加入例外詞，讓後續相同或相近情境降低嚴重度。

適合加入 safe phrase 的情境：

* 程式碼範例
* 教學內容
* 規則引用
* 測試訊息
* 檔名或指令參數
* 非針對他人的討論

---

## 5. 案件系統與人工確認

當 Bot 偵測到疑似違規內容時，會建立案件並送到管理回報頻道。

若啟用：

```env
CREATE_CASE_THREADS=true
```

Bot 會為案件建立 Discord 討論串，方便管理員集中討論同一個案件。

建議流程：

```text
偵測到疑似違規
↓
建立案件
↓
送到管理頻道
↓
建立討論串
↓
管理員確認情境
↓
使用 /case_ignore、/case_resolve、/punish 等指令處理
```

這樣可以避免 Bot 直接依照單次文字判斷做出過度處罰。

---

## 6. 高風險操作需要管理員確認

涉及高風險操作時，例如：

* 禁言
* 踢出
* 封鎖
* 刪除訊息
* 遠端電腦動作
* Bot 關閉

應由管理員確認後執行。

部分指令使用 `confirm:true` 才會真的執行。
如果 `confirm:false`，通常只會 dry-run 或顯示建議。

---

## 7. 管理員與普通成員權限分離

本專案將普通成員指令與管理員指令分開。

普通成員可使用：

* `/status`
* `/ai`
* `/my_warnings`
* `/my_profile`
* 音樂相關指令

管理員可使用：

* 警告與處罰指令
* 案件處理指令
* 規則管理指令
* PC 狀態與遠端動作指令
* Bot 模式與高溫保護設定

管理員判斷可依照 Discord 權限或 `.env` 中設定的 `ADMIN_ROLE_ID`。

---

## 8. public / private 回覆控制

部分 Slash Command 支援 `public` 參數。

```text
public:false
```

代表只讓使用者自己看到。

```text
public:true
```

代表公開顯示在目前頻道。

這可以避免警告紀錄、AI 摘要、使用者狀態等資訊被不必要地公開。

---

## 9. Log 截斷與錯誤保護

Bot 會記錄指令、錯誤與系統事件，但 log 寫入時會限制單筆資料長度，避免過長內容造成 log 爆量。

Logger 本身也採取 best-effort 設計。
如果 log 寫入失敗，不應讓整個 Bot 中斷。

---

## 10. AI 功能的限制與隔離

AI 功能使用本地 Ollama。

需要注意：

* AI 回覆僅作為輔助建議
* 不應把 AI 判斷當作唯一管理依據
* 高風險操作仍需管理員確認
* 使用者互動摘要可能包含個人資料，應避免公開

Bot 也提供不同模式：

```text
normal
eco
emergency
```

在 `eco` 或 `emergency` 模式下，可以降低或停用 AI 與重工作業，保留基本管理與回報功能。

---

## 11. 音樂檔案限制

音樂功能會限制可接受的音檔副檔名，例如：

```text
.mp3
.wav
.flac
.ogg
.m4a
```

並可透過 `.env` 設定：

```env
MUSIC_MAX_TRACKS_PER_USER=20
MUSIC_MAX_FILE_MB=50
```

用來限制每位使用者的歌曲數量與單一檔案大小。

所有上傳音檔的來源、授權與使用權由上傳者自行確認並負責。

Bot 僅提供音檔佇列與播放功能，不代表伺服器、管理員或 Bot 擁有、審核或授權該音檔內容。

---

## 12. PC 狀態與高溫保護

PC 狀態功能可回報：

* CPU 使用率
* RAM 使用量
* 磁碟使用量
* 網路流量
* GPU 狀態
* CPU 溫度

CPU 溫度讀取可搭配 LibreHardwareMonitor。

Bot 可設定高溫保護閾值，例如：

```text
eco_temp
emergency_temp
shutdown_temp
```

當溫度過高時，Bot 可以自動切換低功耗模式或緊急模式，必要時關閉 Bot 自己。

---

## 13. 已知限制

本專案仍有以下限制：

* 規則式違規偵測可能誤判
* AI 摘要可能產生不完整或不準確的判斷
* Discord 權限設定錯誤可能導致指令無法正常執行
* Webhook URL、Bot Token 一旦外洩，需要立即更換
* 實際安全性仍取決於部署者的 `.env`、Discord 權限、伺服器設定與資料管理方式

---

## 14. 建議部署方式

正式部署前建議：

1. 先使用 私有 Repository 測試
2. 確認 `.env` 沒有被 commit
3. 確認 `data/` 與 `logs/` 不會上傳
4. 使用測試 Discord 伺服器驗證功能
5. 開啟必要權限，不要給 Bot 過高權限
6. 管理指令只開放給可信任角色
7. 定期檢查 log 與案件紀錄
8. 若公開專案，請先清除所有私密設定與本機路徑

---

## 15. 總結

本專案的安全設計重點不是完全自動化處罰，而是：

```text
偵測 → 回報 → 討論 → 人工確認 → 執行
```

透過案件系統、討論串、safe phrase、權限分離、private 回覆、log 截斷與 `.env` 分離，降低誤判、資料外洩與過度處罰的風險。
