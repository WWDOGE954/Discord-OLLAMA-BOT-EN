## 程式碼層面的防護設計

除了功能層面的權限限制與管理流程，本專案也在程式碼層面加入一些防護設計，降低誤判、資料損壞、log 爆量與高風險操作失控的可能性。
由於本地模型可能消耗較多 CPU、GPU、RAM 與儲存資源，因此資源分配、溫度監控與資料保存策略也需要被納入考量。


### 1. ToolRouter 與 Discord API 解耦

本專案將管理邏輯集中在 `ToolRouter` 中處理。

`ToolRouter` 本身不直接呼叫 Discord API，而是接收純資料，例如使用者 ID、訊息內容、頻道 ID、案件資料，運算後回傳 `ToolResult`。

Discord 主程式再根據 `ToolResult` 決定：

* 要不要顯示給使用者
* 要不要送到管理頻道
* 要不要建立案件討論串
* 要不要要求管理員確認
* 要不要實際執行禁言或其他操作

這樣做的好處是：

* 違規偵測邏輯不會綁死 Discord 顯示方式
* 核心邏輯比較容易測試
* 高風險動作可以先 dry-run 或標記需要管理員確認
* 未來若要接到其他平台，也能重用部分邏輯

簡化流程如下：

```text
Discord event
↓
整理成純資料
↓
ToolRouter.evaluate_message(...)
↓
回傳 ToolResult
↓
main.py 決定如何顯示、回報或執行
```

---

### 2. 統一回傳格式 ToolResult

`ToolResult` 用來統一工具層回傳結果。

它包含：

```text
ok
tool
message
data
requires_admin_confirm
dry_run
```

其中：

* `ok`：工具是否成功處理
* `message`：給人看的簡短說明
* `data`：機器可讀的詳細資料
* `requires_admin_confirm`：是否需要管理員確認
* `dry_run`：是否只是模擬，尚未實際執行

這讓主程式可以清楚判斷目前結果是「已執行」、「僅建議」、「需人工確認」或「忽略」。

---

### 3. JSON 寫入採用臨時檔替換

資料保存時不會直接覆蓋原始 JSON，而是先寫入 `.tmp` 臨時檔，再用 `replace` 替換原檔。

流程如下：

```text
原始資料檔
↓
先寫入 filename.json.tmp
↓
寫入完成後 replace 成 filename.json
```

這樣可以降低程式中斷、寫入失敗或部分寫入造成 JSON 檔案壞掉的風險。

注意：這不是完整資料庫交易系統，也不能保證任何斷電情境都完全不損壞；但比直接覆蓋原檔更安全，適合作為輕量 JSON 儲存方案。

---

### 4. JSON 讀取失敗時使用預設值

讀取 JSON 時，如果檔案不存在，會回傳預設值。

如果 JSON 格式錯誤，程式也會回傳預設資料，而不是讓整個 Bot 直接崩潰。

這讓 Bot 在 展示、測試或手動編輯 JSON 檔案出錯時，仍能保持基本運作。

---

### 5. 輕量化資料保留策略

本專案使用 JSON / JSONL 作為輕量資料儲存格式。

為了避免長期運行後檔案無限制膨脹，部分資料會限制保存數量，例如：

```text
action_log[-300:]
moderation_cases[-500:]
```

這代表只保留最近一段重要紀錄，避免 展示專案或小型伺服器使用時資料量過大。

這不是完整資料庫設計，但適合低成本、易讀、容易備份的小型 Bot 架構。

---

### 6. Log 截斷與容錯設計

Log 系統會將資料寫成 JSONL，並依照類型分到不同資料夾，例如：

```text
logs/bot/
logs/commands/
logs/music/
logs/errors/
logs/system/
```

為了避免單次訊息、錯誤 traceback 或大型物件造成 log 過大，log 寫入前會先進行截斷：

* 字串限制長度
* list 限制項目數量
* dict 限制鍵值數量
* 不可 JSON 序列化的資料轉成字串

此外，logger 採用 best-effort 設計。
如果 log 寫入失敗，logger 不會讓整個 Bot 中斷。

---

### 7. 高溫保護採用連續命中機制

Bot 的高溫保護不是單次超過溫度就立刻關閉，而是使用：

```text
high_temp_hits
shutdown_hits_required
```

來判斷是否連續多次超過 shutdown 溫度。

這樣可以避免因為短暫溫度尖峰、感測器瞬間異常或讀值跳動，導致 Bot 誤判並直接關閉。

高溫保護大致分為：

```text
eco_temp       → 進入低功耗模式
emergency_temp → 進入緊急模式
shutdown_temp  → 連續命中後關閉 Bot
```

這種設計比較保守，也比較適合長時間運行的本機 Bot。

---

### 8. 音樂檔案名稱與副檔名保護

音樂管理模組會限制允許的音檔副檔名，例如：

```text
.mp3
.wav
.flac
.ogg
.m4a
```

檔名也會經過清理，移除不適合用在檔案路徑中的符號，例如：

```text
\ / : * ? " < > |
```

這可以降低使用者上傳奇怪檔名造成路徑錯誤或檔案管理混亂的風險。


---

### 9. 管理員訊息只回報不自動處罰

當具管理權限者觸發違規規則時，系統會採取較保守的處理方式。

管理員訊息可以被偵測與回報，但不會直接套用自動處罰。
這可以避免管理員在測試、引用、教學或處理案件時誤觸規則而被 Bot 自動處理。

---

### 10. 自動處罰前先判斷情境

Bot 並不是只要看到禁用詞就一定處罰。

它會考慮：

* 是否命中 `safe_phrases`
* 是否為針對性攻擊
* 是否提及其他使用者
* 是否屬於 spam 或高風險分類
* 使用者是否為管理員
* 是否在短時間內多次觸發

如果內容疑似違規但缺少明確攻擊對象，可能只建立案件並回報，而不是直接處罰。

---

## 小結

本專案的程式碼防護核心是：

```text
資料與平台解耦
↓
結果統一封裝
↓
高風險操作 dry-run / confirm
↓
JSON 寫入容錯
↓
log 截斷與分類
↓
溫度保護
↓
人工確認優先
```

整體設計目標不是讓 Bot 完全自動取代管理員，而是建立一個比較安全、可追蹤、可人工介入的管理輔助流程。
同時也讓 Discord 事件、管理邏輯與本地語言模型之間保持較清楚的邊界，避免 AI 功能直接介入高風險操作。
