## 代码层面的防护设计

除了功能层面的权限限制与管理流程，本项目也在代码层面加入一些防护设计，降低误判、数据损坏、log 爆量与高风险操作失控的可能性。
由于本地模型可能消耗较多 CPU、GPU、RAM 与存储资源，因此资源分配、温度监控与数据保存策略也需要被纳入考量。


### 1. ToolRouter 与 Discord API 解耦

本项目将管理逻辑集中在 `ToolRouter` 中处理。

`ToolRouter` 本身不直接呼叫 Discord API，而是接收纯数据，例如用户 ID、消息内容、频道 ID、案件数据，运算后返回 `ToolResult`。

Discord 主程序再根据 `ToolResult` 决定：

* 要不要显示给用户
* 要不要发送到管理频道
* 要不要创建案件讨论串
* 要不要要求管理员确认
* 要不要实际运行禁言或其他操作

这样做的好处是：

* 违规检测逻辑不会绑定在 Discord 显示方式
* 核心逻辑比较容易测试
* 高风险动作可以先 dry-run 或标记需要管理员确认
* 未来如果要接入其他平台，也能重用部分逻辑

简化流程如下：

```text
Discord event
↓
整理成纯数据
↓
ToolRouter.evaluate_message(...)
↓
返回 ToolResult
↓
main.py 决定如何显示、报告或运行
```

---

### 2. 统一返回格式 ToolResult

`ToolResult` 用来统一工具层返回结果。

它包含：

```text
ok
tool
message
data
requires_admin_confirm
dry_run
```

其中：

* `ok`：工具是否成功处理
* `message`：给人看的简短说明
* `data`：机器可读的详细数据
* `requires_admin_confirm`：是否需要管理员确认
* `dry_run`：是否只是模拟，尚未实际运行

这让主程序可以清楚判断目前结果是「已运行」、「仅建议」、「需人工确认」或「忽略」。

---

### 3. JSON 写入采用临时文件替换

数据保存时不会直接覆盖原始 JSON，而是先写入 `.tmp` 临时文件，再用 `replace` 替换原档。

流程如下：

```text
原始数据文件
↓
先写入 filename.json.tmp
↓
写入完成后使用 replace 替换成 filename.json
```

这样可以降低程序中断、写入失败或部分写入造成 JSON 文件坏掉的风险。

注意：这不是完整数据库交易系统，也不能保证任何断电场景都完全不损坏；但比直接覆盖原档更安全，适合作为轻量 JSON 存储方案。

---

### 4. JSON 读取失败时使用默认值

读取 JSON 时，如果文件不存在，会返回默认值。

如果 JSON 格式错误，程序也会返回预设数据，而不是让整个 Bot 直接中断。

这让 Bot 在展示、测试或手动编辑 JSON 文件出错时，仍能保持基本运作。

---

### 5. 轻量化数据保留策略

本项目使用 JSON / JSONL 作为轻量数据存储格式。

为了避免长期运行后文件无限制膨胀，部分数据会限制保存数量，例如：

```text
action_log[-300:]
moderation_cases[-500:]
```

这代表只保留最近一段重要记录，避免展示项目或小型服务器使用时数据量过大。

这不是完整数据库设计，但适合低成本、易读、容易备份的小型 Bot 架构。

---

### 6. Log 截断与容错设计

Log 系统会将数据写成 JSONL，并按照类型分到不同文件夹，例如：

```text
logs/bot/
logs/commands/
logs/music/
logs/errors/
logs/system/
```

为了避免单条消息、错误 traceback 或大型物件造成 log 过大，log 写入前会先进行截断：

* 字符串限制长度
* list 限制项目数量
* dict 限制键值数量
* 不可 JSON 序列化的数据转成字符串

此外，logger 采用 best-effort 设计。
如果 log 写入失败，logger 不会让整个 Bot 中断。

---

### 7. 高温保护采用连续命中机制

Bot 的高温保护不是单次超过温度就立刻关闭，而是使用：

```text
high_temp_hits
shutdown_hits_required
```

来判断是否连续多次超过 shutdown 温度。

这样可以避免因为短暂温度尖峰、感测器瞬間异常或读值跳动，導致 Bot 误判并直接关闭。

高温保护大致分为：

```text
eco_temp       → 进入低功耗模式
emergency_temp → 进入紧急模式
shutdown_temp  → 连续命中后关闭 Bot
```

这種设计比较保守，也比较适合长时间运行的本机 Bot。

---

### 8. 音乐文件名称与扩展名保护

音乐管理模组会限制允許的音频文件扩展名，例如：

```text
.mp3
.wav
.flac
.ogg
.m4a
```

文件名也会經过清理，移除不适合用在文件路径中的符号，例如：

```text
\ / : * ? " < > |
```

这可以降低用户上传奇怪文件名造成路径错误或文件管理混乱的风险。


---

### 9. 管理员消息只报告不自动处罚

当具管理权限者触发违规规则时，系统会採取较保守的处理方式。

管理员消息可以被检测与报告，但不会直接套用自动处罚。
这可以避免管理员在测试、引用、教学或处理案件时误触规则而被 Bot 自动处理。

---

### 10. 自动处罚前先判断场景

Bot 并不是只要看到禁用词就一定处罚。

它会考虑：

* 是否命中 `safe_phrases`
* 是否为针对性攻击
* 是否提及其他用户
* 是否属于 spam 或高风险分类
* 用户是否为管理员
* 是否在短时间内多次触发

如果内容疑似违规但缺少明确攻击对象，可能只创建案件并报告，而不是直接处罚。

---

## 小结

本项目的代码防护核心是：

```text
数据与平台解耦
↓
结果统一封装
↓
高风险操作 dry-run / confirm
↓
JSON 写入容错
↓
log 截断与分类
↓
温度保护
↓
人工确认優先
```

整体设计目标不是让 Bot 完全自动取代管理员，而是创建一个比较安全、可追踪、可人工介入的管理辅助流程。
同时也让 Discord 事件、管理逻辑与本地语言模型之間保持较清楚的边界，避免 AI 功能直接介入高风险操作。
